# План разработки: сервис бронирования звонков

## Контекст

Проект — упрощённый аналог Cal.com: один владелец календаря, типы событий,
генерируемые слоты, бронирования без авторизации. Архитектура:
`main.tsp` (контракт, источник истины) → OpenAPI → типы фронта; Express-бэкенд
с in-memory store; React 19 + Vite + React Query; E2E на Playwright.

Цель плана — задать приоритизированный бэклог для агента: сначала закрыть
реальные дефекты в текущем коде (генерация слотов, таймзоны, валидация
бронирований), затем нарастить функциональность поверх MVP. Согласованные рамки:
**сбалансированный роадмап**, **допустимы инфраструктурные изменения**
(персистентность, пагинация и т.п.), **каждый пункт сопровождается E2E-тестом**
(Playwright), по принятому в проекте стилю (`frontend/tests/**`).

Важно: при любых изменениях контракта сперва правится `main.tsp`, затем
`npx tsp compile . --emit=@typespec/openapi3` и `npm run api:generate`
(см. [AGENTS.md](AGENTS.md)). `tsp-output/` и
[frontend/src/api/generated.ts](frontend/src/api/generated.ts) не редактируются вручную.

## Часть 1. Баги (исправление причины, не симптома)

### BUG-1 (P0) — Перекрывающиеся/некорректные слоты при длительности, не кратной 60
**Где:** [backend/src/utils/slots.ts:30-41](backend/src/utils/slots.ts#L30-L41)
**Причина:** слоты генерируются вложенными циклами «по часам → внутри часа от :00
с шагом durationMinutes», а внутренний цикл сбрасывает минуты к 0 в начале
каждого часа. Для длительностей 45/90/120 мин (диапазон контракта 1–480) слоты
наезжают друг на друга (напр. 9:45–10:30 и 10:00–10:45) либо генерируются неверно.
**Фикс:** переписать на единый непрерывный проход в пределах рабочего окна дня —
от `workStartHour` до `workEndHour` шагом `durationMinutes`, без вложенного
почасового цикла; слот добавляется только если `currentEnd` ≤ конца окна.
**E2E:** создать тип события 45 мин → проверить, что соседние слоты не
пересекаются и шаг равен длительности.

### BUG-2 (P0) — Слоты в прошлом показываются доступными
**Где:** [backend/src/utils/slots.ts:9-13](backend/src/utils/slots.ts#L9-L13)
**Причина:** `startDate` сбрасывается к началу дня, прошедшие слоты текущего дня
возвращаются с `isAvailable: true`; фронт их не фильтрует
([PublicEventTypeDetails.tsx:109](frontend/src/pages/public/PublicEventTypeDetails.tsx#L109)).
**Фикс:** в `generateSlots` отсекать слоты с `currentStart <= now` (для сегодняшней
даты). Это же закрывает возможность брони задним числом на бэке.
**E2E:** запросить слоты на сегодня → убедиться, что слотов раньше текущего времени нет.

### BUG-3 (P1) — Игнорируется таймзона владельца
**Где:** [backend/src/utils/slots.ts:30-32](backend/src/utils/slots.ts#L30-L32) (`setUTCHours`),
seed `timezone: "Europe/Moscow"` ([store/seed.ts:10](backend/src/store/seed.ts#L10)).
**Причина:** рабочие часы 9–18 жёстко заданы в UTC, поле `owner.timezone` не
используется. Для московского владельца «рабочий день» фактически 12:00–21:00 MSK.
**Фикс:** генерировать рабочее окно относительно `owner.timezone` (через
`Intl.DateTimeFormat`/смещение зоны), отдавать `startTime/endTime` по-прежнему в UTC
(ISO). Фронт уже корректно форматирует в локали — но для предсказуемости
рассмотреть показ таймзоны организатора в UI бронирования.
**E2E:** для владельца с известной зоной проверить, что первый слот дня
соответствует ожидаемому локальному часу.

### BUG-4 (P1) — Бронирование принимает произвольный slotId
**Где:** [backend/src/routes/public.ts:45-76](backend/src/routes/public.ts#L45-L76),
валидатор [validators/bookings.ts:4](backend/src/validators/bookings.ts#L4) (`slotId` = просто строка).
**Причина:** при `POST /public/bookings` проверяется только отсутствие
конфликтующей брони. Можно забронировать несуществующий/несогласованный слот
(вне рабочих часов, не по сетке, в прошлом). Фронт уже ожидает ответ 422
([PublicEventTypeDetails.tsx:101](frontend/src/pages/public/PublicEventTypeDetails.tsx#L101)),
но бэк его не возвращает — обработчик мёртв.
**Фикс:** валидировать slotId на сервере — разобрать `eventTypeId_ISO`, проверить
существование типа события, что время попадает в сетку `generateSlots`
(переиспользовать общую функцию построения сетки) и не в прошлом; иначе 422
`VALIDATION`. Отразить ответ 422 в контракте `main.tsp` для `POST /public/bookings`.
**E2E:** отправить бронь с заведомо невалидным slotId → ожидать 422 и тост
«Некорректный слот».

### BUG-5 (P2) — Фильтр админских броней по несуществующему eventTypeId возвращает всё
**Где:** [backend/src/routes/admin.ts:81-87](backend/src/routes/admin.ts#L81-L87)
**Причина:** если `eventTypeId` передан, но тип не найден, блок `if (et)`
пропускается и фильтрация не применяется — отдаются все брони.
**Фикс:** фильтровать по префиксу slotId напрямую по переданному `eventTypeId`
(или возвращать пустой список при неизвестном id), не завязываясь на наличие типа.
**E2E:** запрос `GET /admin/bookings?eventTypeId=<неизвестный>` → пустой список.

## Часть 2. Новые фичи

### FEAT-1 (P1) — Перенос бронирования (reschedule)
Гость/админ переносит бронь на другой свободный слот без отмены и повторного
создания. Контракт: `POST /public/bookings/{id}/reschedule` (+ админский аналог),
тело `{ slotId }`, проверки занятости/валидности слота (переиспользовать BUG-4).
UI: кнопка «Перенести» на странице подтверждения и в админ-детали брони.
**E2E:** бронь → перенос на другой слот → старый слот снова свободен, новый занят.

### FEAT-2 (P1) — Настраиваемые рабочие часы и доступность
Вынести жёстко зашитые 9–18 в настройки (на уровне владельца или типа события):
`workStartHour`, `workEndHour`, рабочие дни недели. Контракт + админ-UI редактирования.
Снимает магические константы из [slots.ts:17-18](backend/src/utils/slots.ts#L17-L18).
**E2E:** изменить рабочие часы → набор слотов меняется соответственно.

### FEAT-3 (P2) — Буфер между встречами
Поле `bufferMinutes` у типа события; учитывать буфер при генерации сетки и при
проверке конфликтов соседних броней.
**E2E:** при заданном буфере соседние слоты разнесены на duration + buffer.

### FEAT-4 (P2) — Подтверждение брони по email (заглушка/лог)
Сервис уведомлений (интерфейс + dev-реализация, пишущая в лог/в память),
вызов при создании/отмене/переносе. Без внешнего SMTP — точка расширения.
**E2E:** после брони уведомление зафиксировано (через тестовый эндпоинт/лог).

### FEAT-5 (P2) — Пагинация и поиск в списках админки
`GET /admin/bookings` и список типов событий: query `page`/`pageSize`, поиск по
имени гостя/типу. UI-пагинация в [BookingsList.tsx](frontend/src/pages/admin/BookingsList.tsx).
**E2E:** при N+1 бронях видна вторая страница; поиск фильтрует результат.

### INFRA-1 (P1) — Персистентность стора
Заменить in-memory Map на сохраняемое хранилище (файловый JSON или SQLite),
сохранив текущий интерфейс [store/index.ts](backend/src/store/index.ts), чтобы

персистентный слой.
**Проверка:** создать бронь → перезапустить бэк → бронь на месте.

Поднять Vitest в `backend/` и покрыть `generateSlots` (кратность, прошлое,
